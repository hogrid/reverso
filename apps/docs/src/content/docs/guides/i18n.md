# Internationalization (i18n)

Reverso supports multiple languages out of the box.

## Configuration

Enable i18n in `reverso.config.ts`:

```typescript
export default {
  i18n: {
    // Default locale
    defaultLocale: 'en',

    // Available locales
    locales: ['en', 'pt', 'es', 'fr', 'de'],

    // Locale display names (optional)
    localeNames: {
      en: 'English',
      pt: 'Português',
      es: 'Español',
      fr: 'Français',
      de: 'Deutsch',
    },
  },
} satisfies ReversoConfig;
```

## How It Works

1. **Content is stored per locale** - Each field value has a locale
2. **Admin shows locale selector** - Editors choose which language to edit
3. **API returns locale-specific content** - Pass locale parameter

## Admin Panel

When i18n is enabled:

- **Locale selector** appears in the header
- **Content is saved separately** for each locale
- **Fallback indicator** shows if content falls back to default

## Fetching Localized Content

### REST API

```bash
# Get content in Portuguese
GET /api/reverso/content/home.hero.title?locale=pt

# Get all page content in Spanish
GET /api/reverso/content/page/home?locale=es
```

### Code Example

```typescript
async function getContent(path: string, locale: string) {
  const response = await fetch(
    `/api/reverso/content/${path}?locale=${locale}`
  );
  return response.json();
}

// Usage
const title = await getContent('home.hero.title', 'pt');
```

## Frontend Integration

### Next.js App Router

```tsx
// app/[locale]/page.tsx
import { getContent } from '@reverso/client';

export default async function HomePage({
  params: { locale },
}: {
  params: { locale: string };
}) {
  const content = await getContent('home', locale);

  return (
    <h1>{content.hero.title}</h1>
  );
}

// app/[locale]/layout.tsx
export function generateStaticParams() {
  return [
    { locale: 'en' },
    { locale: 'pt' },
    { locale: 'es' },
  ];
}
```

### Next.js Pages Router

```tsx
// pages/[locale]/index.tsx
import { getContent } from '@reverso/client';

export async function getStaticProps({ params }) {
  const content = await getContent('home', params.locale);

  return {
    props: { content },
    revalidate: 60,
  };
}

export async function getStaticPaths() {
  return {
    paths: [
      { params: { locale: 'en' } },
      { params: { locale: 'pt' } },
    ],
    fallback: false,
  };
}
```

## Content Fallback

When content doesn't exist for a locale, Reverso can fall back to the default locale:

```typescript
// reverso.config.ts
export default {
  i18n: {
    defaultLocale: 'en',
    locales: ['en', 'pt', 'es'],
    // Enable fallback to default locale
    fallback: true,
  },
};
```

With fallback enabled:
1. Request content in `pt`
2. If not found, return content in `en`
3. If still not found, return null

## Translation Workflow

### For Content Editors

1. Select locale in admin header
2. Edit content for that locale
3. Switch locales to translate
4. Published status is per-locale

### Recommended Workflow

1. **Create content in default locale first**
2. **Mark as ready for translation**
3. **Translators work on each locale**
4. **Review and publish per locale**

## Markers with i18n

Markers work the same way - the locale is determined at runtime:

```tsx
// The marker doesn't change
<h1 data-reverso="home.hero.title">Welcome</h1>

// Content is fetched with locale parameter
const content = await getContent('home.hero.title', userLocale);
```

## MCP Integration

Use MCP tools with locale:

```typescript
// Get content in specific locale
await mcp.call('content_get_content', {
  path: 'home.hero.title',
  locale: 'pt',
});

// Update content in specific locale
await mcp.call('content_update_content', {
  path: 'home.hero.title',
  value: 'Bem-vindo',
  locale: 'pt',
});
```

## Best Practices

### 1. Start with Default Locale

Create and finalize content in your default locale before translating.

### 2. Use Translation Keys

For static UI elements, use a translation library (like `next-intl` or `react-i18next`). Use Reverso for dynamic content.

### 3. Consider URL Structure

Choose a URL strategy:

| Strategy | Example |
|----------|---------|
| Subdirectory | `/en/about`, `/pt/about` |
| Subdomain | `en.site.com`, `pt.site.com` |
| Query param | `/about?locale=pt` |

### 4. Handle RTL Languages

For Arabic, Hebrew, etc., ensure your UI supports RTL.

### 5. Locale-Specific Media

Some images may need localization (screenshots with text, etc.). Use separate image fields or a naming convention.

## API Reference

### Get Content with Locale

```bash
GET /api/reverso/content/:path?locale=pt
```

### Update Content with Locale

```bash
PUT /api/reverso/content/:path

{
  "value": "Translated content",
  "locale": "pt"
}
```

### Bulk Update with Locale

```bash
POST /api/reverso/content/bulk

{
  "locale": "pt",
  "updates": [
    { "path": "home.hero.title", "value": "Título" },
    { "path": "home.hero.subtitle", "value": "Subtítulo" }
  ]
}
```
